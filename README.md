# Индивидуальное задание – Бэкенд для многопользовательской игры

В этом задании вы самостоятельно спроектируете и реализуете серверную часть игры **Frog Chess** — пошаговой игры для двух игроков на доске 8×8. В качестве ориентира вам предоставлен готовый фронтенд.

[Frogchess](https://xivol.github.io/greenswamp/indiv/frogchess/frogchess)

---

## Правила игры — Frog Chess

Сервер является **единственным источником истины** в отношении состояния игры и проверки правил. Прочитайте правила внимательно — корректная реализация на сервере составляет значительную часть оценки.

[Frog Chess Rules](rules-bgg434812.pdf)

Важные моменты для реализации:

- Игра всегда проходит **1 на 1**. Это упрощает реализацию.

- **Расстановка** — Стартовую расстоновку генерирует сервер. У каждого игрока **18 лягушек** своего цвета. Когда оба игрока подключились, сервер **случайным образом** перемешивает и распределяет все 36 лягушек по белым клеткам, после чего рассылает начальное состояние доски обоим игрокам. Никакой ручной расстановки нет — игра начинается немедленно. Первым ходит игрок, создавший игру.

- **Только на своём самом первом ходу**, до совершения прыжка, игрок вправе убрать с доски **любую** лягушку — свою или чужую. Это обязательно; игрок не может сразу перейти к прыжку. Повторно воспользоваться этим правом нельзя.

- Если игрок не может совершить прыжок, он должен **пропустить ход**.

- **Конец игры** — Игра заканчивается, когда **оба** игрока подряд не смогли прыгнуть — то есть на доске не осталось ни одного допустимого прыжка ни для кого. Победу получает тот, кто совершил **последний прыжок** в партии.

## Что нужно реализовать

Ваш бэкенд должен поддерживать как минимум следующее:

- **Создание и вход в игру.** Игрок создаёт новую игру и получает идентификатор для передачи сопернику. Второй игрок подключается по этому идентификатору. Когда оба на месте, сервер генерирует случайную расстановку и рассылает её обоим.
- **Первый ход — убирание лягушки.** Перед своим первым прыжком каждый игрок может один раз убрать с доски любую лягушку. Сервер разрешает это только до первого прыжка данного игрока и не более одного раза.
- **Игровой процесс — прыжки.** Игроки отправляют ходы (одиночные и цепочечные прыжки). Сервер проверяет допустимость каждого прыжка в цепочке, убирает перепрыгнутых лягушек и уведомляет обоих игроков в реальном времени.
- **Пропуск хода.** Если у игрока нет допустимых прыжков, его ход пропускается, ход переходит к сопернику. Сервер должен определять это автоматически и уведомлять обоих игроков.
- **Механику болота.** Сервер отслеживает, заканчивает ли лягушка ход в болоте, и убирает её в таком случае.
- **Определение победителя.** Игра заканчивается, когда оба игрока подряд не смогли прыгнуть. Победа присуждается тому, кто совершил последний прыжок в партии.
- **Обработку отключений.** Если игрок отключается, оставшийся получает уведомление. Отключившемуся даётся разумное окно для переподключения (вы определяете длину) — после его истечения победа засчитывается сопернику.

Всё остальное — на ваше усмотрение.

## Архитектурное решение: REST, SignalR или гибрид

Это главный вопрос проектирования в задании. Вы обязаны ответить на него явно в README.

Три реалистичных варианта:

**А — Чистый SignalR.** Вся коммуникация клиента с сервером проходит через Hub. Ходы передаются вызовами методов Hub, обновления — Hub-событиями. Просто, низкая задержка, ориентировано на постоянное соединение.

**Б — Гибридный подход.** Команды (ходы, вход в игру, сдача) — через REST-эндпоинты. События (обновление состояния, подключение противника, конец игры) — через SignalR. REST даёт стандартную HTTP-семантику и упрощает тестирование; SignalR берёт на себя рассылку уведомлений.

**В — Чистый REST с polling.** Постоянного соединения нет, клиент периодически опрашивает сервер. Просто реализовать, плохо подходит для игры в реальном времени — допустимо только при наличии убедительного обоснования в README с честным разбором недостатков.

Единственно верного ответа нет. Хорошо обоснованный и корректно реализованный вариант Б получит ту же оценку, что и хорошо обоснованный и корректно реализованный вариант А. Оценка снижается, если выбор сделан без объяснения причин или если реализация не соответствует заявленному подходу.

В README должен быть раздел **«Архитектурное решение»** (один-два абзаца), в котором указано: какой вариант выбран, почему, от чего вы отказались, не выбрав альтернативы, и был ли этот компромисс оправдан для данной задачи.

## Требования к тестам

Обязательно протестировать:

- **Генерацию расстановки** — все 36 клеток заняты, ровно 18 лягушек каждого цвета, ни одна клетка не повторяется; при каждом запуске расстановка статистически отличается.
- **Первый ход — убирание лягушки** — разрешено до первого прыжка, только один раз; попытка убрать лягушку после уже совершённого прыжка отклоняется; повторная попытка отклоняется.
- **Валидацию прыжков** — допустимые принимаются, недопустимые отклоняются. Покройте как минимум: прыжок через свою лягушку, прыжок через чужую, цепочку из двух прыжков, прыжок не по прямой (недопустим), приземление на занятую клетку (недопустимо), прыжок через несколько клеток (недопустимо), ход не в свою очередь (недопустим).
- **Механику болота** — лягушка, завершившая ход в болоте, убирается с доски; лягушка, прыгнувшая из болота обратно на белую клетку, остаётся.
- **Пропуск хода и конец игры** — если у активного игрока нет ходов, ход корректно передаётся сопернику; если оба игрока подряд не могут прыгнуть, игра завершается и победа присуждается тому, кто прыгнул последним.
- **Конкурентность** — два одновременно поданных хода в одной игре не повреждают состояние.

Количество тестов и выбор фреймворка — на ваше усмотрение. Важно, чтобы тесты были содержательными и охватывали перечисленные области.

## Нефункциональные требования

- **Целевой фреймворк:** .NET 8-10
- **База данных не нужна:** только состояние в памяти — перезапуск сервера сбрасывает все игры, это допустимо
- **Аутентификация не требуется:** идентификация игрока по соединению или по токену/имени на ваш выбор — главное, задокументировать
- **CORS:** настройте так, чтобы фронтенд (или ваш тестовый клиент) мог подключаться в режиме разработки
- **Логирование:** логов должно быть достаточно, чтобы по ним можно было проследить жизненный цикл игры без чтения кода

## Советы

**Сначала модель, потом проводка.** Добейтесь, чтобы представление доски, валидатор прыжков (включая цепочки и болото) и логика конца игры работали и были покрыты тестами, прежде чем касаться SignalR или HTTP. Это чистые C#-классы без инфраструктурных зависимостей — их быстро написать и быстро проверить.

**Продумайте стратегию асинхронности заранее.** Несколько соединений могут вызывать методы одновременно. Решите сразу, как будете защищать общее состояние — локом на игру, `SemaphoreSlim`, каналом — и применяйте это последовательно. Тест на конкурентность вскроет любые пробелы.

**Переподключение сложно, если не планировать его заранее.** Подумайте, какой информации достаточно, чтобы связать новое соединение с существующей игровой сессией. Обычно хватает имени игрока и идентификатора игры, но подумайте, что будет, если два разных человека введут одно и то же имя.

**В гибридном подходе есть проблема идентификации.** Если команды идут через REST, HTTP-запрос не несёт встроенной идентичности, как соединение SignalR. Нужно что-то придумать — даже простой непрозрачный токен игрока, выданный при входе, подойдёт. Не превращайте это в полноценную аутентификацию; случайный GUID вполне достаточен.

**Документируйте по ходу работы.** Документация API — лёгкий пункт, если писать её параллельно с кодом. Если оставить на последний вечер перед сдачей, превратится в каторгу.
